# AGENTS Instructions

## コードコメント方針
- 新規生成・修正するコードには、要点が分かる日本語コメントを適宜追加すること。
- コメントは処理意図・設計判断・注意点を優先し、冗長な逐語説明は避けること。
- 既存コードを大きく変更した場合は、変更箇所の入口となる部分に日本語コメントを付けること。

## 命名規則
- 変数名・関数名・型名・ファイル名は、原則として日本語のローマ字（ヘボン式）で命名すること。
- 英単語ベースの既存APIやライブラリ都合を除き、`customer` のような英訳語は新規導入しないこと。
- カタカナ語で一般的な英語綴りが定着している語は、ローマ字化より英語綴りを優先すること（例: `sanpuru` ではなく `sample`）。
- 命名で「入力」「出力」を含む場合は、`nyuryoku` / `syuturyoku` ではなく `input` / `output` を使用すること。
- 命名で「内部」を含む場合は、`naibu` ではなく `internal` を使用すること。
- 例外的な省略を許可する。代表例:
  - `code` -> `cd`
  - `kubun` -> `kbn`
- 省略を使う場合は、同一ドメイン内で表記を統一し、混在（例: `code` と `cd` の併用）を避けること。

## スタイル配置方針
- `src/styles.scss` などの共通スタイルには、複数画面で再利用できる汎用スタイルだけを記述すること。
- 画面レイアウト、画面固有の配色、画面専用クラスなど、その画面でしか使わないスタイルは各コンポーネントの `styleUrls` 側に記述すること。
- 共通スタイルへ移す場合は、「別画面でも同じ責務で再利用されるか」を判断基準にし、単なる流用見込みだけでは共通化しないこと。
- 既存の画面固有スタイルが共通スタイルに置かれている場合は、新規修正時に可能な範囲でコンポーネントスタイルへ戻すことを優先すること。
- ビルド制約などで例外的に共通スタイルへ配置する場合は、その理由を対象箇所の近くに日本語コメントで明記すること。
